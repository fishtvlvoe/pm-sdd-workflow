# PM-SDD 術語對照表

給沒用過 AI 或不熟技術術語的 PM 查。跟《PM-SDD 工作模式》、《AI 協作提示詞:需求釐清四步驟》搭配使用。

| 術語(英文/縮寫) | 白話翻譯 | 什麼時候會用到 | 對應 SOP 哪個 Phase |
|---|---|---|---|
| FRD(Functional Requirement Document,功能需求規格書) | 把客戶模糊的需求,寫成工程師可以直接照做的正式說明書。全案的「唯一標準」,客戶、PM、工程師看的都是同一份 | 交接給工程師之前,要有一份大家都認的規格書 | Phase 4 |
| Tech Spec(技術規格書) | 工程師依照 FRD,寫出「技術上要怎麼實現」的文件,例如用什麼架構、資料庫怎麼設計 | 工程師開始寫程式之前 | Phase 5 |
| Test Spec(測試規格書) | 工程師依照 FRD 的驗收標準,寫出「要怎麼測試才算過關」的文件 | 工程師寫完功能,要驗證有沒有做對的時候 | Phase 5 |
| AC(Acceptance Criteria,驗收標準) | 每個功能「在什麼情況下應該發生什麼結果」的具體條件,工程師跟客戶都用這個判斷功能有沒有做對 | 寫 FRD 的時候一定要有,驗收的時候拿出來對照 | Phase 4、Phase 6 |
| SIT(System Integration Testing,系統整合測試) | 把所有功能兜在一起,當成一個完整系統來測試,確認彼此不會互相打架 | 工程師做完個別功能後,上線前的最後一關測試 | Phase 5(尾聲)接 Phase 6(前置) |
| 雛形 / Vibe Coding | 先做一個看得到、點得到的畫面草稿或 Demo 給客戶看,不是正式能上線的系統,重點是先對齊方向 | 需求還沒完全定案,但客戶需要「看到東西」才能確認方向對不對的時候 | Phase 3 |
| SDD(Spec-Driven Development,規格驅動開發) | 跟 Vibe Coding 相對的做法:做完雛形之後不是結案,而是把雛形變成正式規格書,讓需求持續被追蹤、被驗證 | 雛形對齊之後,決定「接下來怎麼把方向變成可以動工的規格」的時候 | Phase 4 之後整個流程的精神 |
| User Story(使用者故事) | 用「身為[角色],我想要[做什麼],以便[達成什麼目的]」這種固定句型,描述一個功能是誰、為了什麼理由需要 | 寫 FRD 的時候,確保每個功能都有明確的使用情境,不是憑空生出來的 | Phase 4 |
| 邊界例外 / Edge Case | 「不是正常流程,但實際可能發生」的情況,例如「負責人員請假當天已經有預約怎麼辦」。這是最容易被忽略、卻最容易上線後爆炸的地方 | 寫 FRD 的時候一定要逐一列出,想不到答案也要標注「⚠️ 待補」,不能跳過 | Phase 4 |
| 1+1 團隊模式 | 一個 PM(負責搞懂客戶要什麼、把需求變成規格)搭配一個全端工程師(負責把規格變成能動的系統),分工清楚,PM 不用會寫代碼,工程師也不用自己猜需求 | 描述整套 SOP 背後的團隊分工方式,適合規模較小、不養整個大團隊的接案案子 | 貫穿 Phase 0 到 Phase 6 |
| 收斂 Loop(收斂迴圈) | FRD 初稿寫完後不能直接拿去用,要反覆跟客戶對、跟 AI 補漏洞,至少跑兩輪,確定沒有漏洞或誤解才算定案 | FRD 初稿出來之後,交給工程師之前 | Phase 4 |
| RFQ(Request for Quotation,報價需求書) | 客戶或 PM 整理出來、用來估價或發包的需求文件,通常還不是正式規格書,比較像 FRD 還沒模板化前的雛形版本 | 案子還在報價或簽約階段,還沒進入正式規格收斂的時候。若客戶初談時就已經帶著自己的 RFQ,也可能在 Phase 4 被拿出來當 FRD 的參考底稿,不是只會出現在最前面 | Phase 0 到 Phase 1 之間(也可能延伸到 Phase 4) |
| 雛形功能分析 | 比對需求清單跟雛形/Demo,抓出「可直接用/需新做/被漏掉」的分析步驟 | 雛形對齊完成後、準備寫 FRD 之前 | Phase 3 |
| 交叉引用探測 | 寫新的功能模組規格前,先檢查跟其他已寫好模組有沒有銜接點或衝突 | 案子有多個功能模組、要寫 FRD 時 | Phase 4 |
| 追溯矩陣(Traceability Matrix) | 用表格對照每個需求有沒有對應的測試案例,確保沒有漏測。這是 RD 寫 Test Spec 時用的工具,PM 不用自己做,但驗收時可以看這張表確認覆蓋率 | Phase 5 驗收,RD 可能會提供這張表給你看 | Phase 5 |
| 1+1 SDD 路徑二 | 團隊導入健檢清單裡定義的一種團隊組成(PM×1 + RD×1),接案 PM 常見的搭配就屬於這一種 | 評估新案子適不適合走這套 SOP 時 | 對應健檢清單,非 6 個 Phase 之一 |
