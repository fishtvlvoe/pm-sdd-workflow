# PM-SDD 團隊導入健檢清單

## 這份清單在做什麼

這不是「一個團隊要不要導入 SDD 方法論」的評估工具。這份清單是給固定用《PM-SDD 工作模式》這套 SOP 的個人接案 PM,每次接到新案子、配到新的外包工程師(RD)時,用來快速判斷「這個案子 + 這個 RD 組合」適不適合走 1+1 SDD 模式,還是應該調整做法。建議在正式開始跑六個 Phase 之前,先過一次這份清單。

## 三種路徑對照

| 路徑 | 人數配置 | 前提條件 |
|---|---|---|
| 路徑一 | 1 人全能型 | 需同時掌握產品 + 技術 |
| 路徑二(接案 PM 常見走這條) | 1+1 SDD(PM×1 + RD×1) | PM:能懂系統基本運作、能讀懂資料庫結構、能寫清楚的 Prompt/規格。RD:以 FRD 為施工依據、不自行腦補解讀。協作:固定的交接與驗收節奏(例如每週一次對齊 + Demo 驗收) |
| 路徑三 | 中型團隊(5-10 人) | Sprint 節奏穩定、完成定義(Definition of Done)明確、RD 以驗收標準(AC)為準、PM/TPM 熟悉雛形工具、需求交接要有書面記錄 |

## 暫停警訊

案子進行中,如果出現以下任何一個可觀察的具體信號(不是「感覺不太對」這種模糊感受),要提高警覺:

| 信號 | 說明 |
|---|---|
| 功能做出來的速度比驗收的速度快 3 倍以上 | 代表規格品質沒跟上開發速度,規格可能太粗糙 |
| 驗收時才發現流程漏掉、要重新討論 | 代表前面規劃不夠完整 |
| RD 越來越常說「我以為需求是這樣」 | 代表需求溝通在中間出現落差 |
| PM 只有在最後驗收才出現,起頭定義需求時沒參與 | 代表分工角色錯位,PM 的核心工作就是把好起頭這關 |

## 出現警訊之後怎麼做

出現任兩項以上警訊,建議暫停案子往下走,回頭補強 FRD 或重新對齊需求,不要硬著頭皮繼續開發。
