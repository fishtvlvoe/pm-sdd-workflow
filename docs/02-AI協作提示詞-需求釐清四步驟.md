# AI 協作提示詞:需求釐清四步驟

這份文件是要貼給 AI Agent 用的指令(ChatGPT、Claude、Gemini 或任何對話式 AI 都可以),不是給人看的說明書。裡面有四段可以直接複製貼上的完整 Prompt,分別對應《PM-SDD 工作模式》的 Phase 1(場景引導)、Phase 2(AI 落地可能性轉譯)、Phase 3(雛形功能分析)、Phase 4(需求規格書產出)。

用法:把逐字稿或前一步的輸出貼在對應的 Prompt 後面(或依照 Prompt 裡「輸入」段落的說明附上資料),整段丟給 AI 執行即可,不需要額外調整格式。四段要依序執行,不要跳過順序;步驟三本身還分兩階段,階段一跟階段二之間也要停下來確認才能繼續,不能一次做完就把結果丟出來。

---

## 步驟一(對應 Phase 1:場景引導)

```
你是一位資深的需求分析顧問。我會給你一份客戶會議的逐字稿或筆記,你的任務不是摘要會議內容,而是「反過來抓出客戶還沒講清楚的地方」。

【輸入】
客戶會議逐字稿或筆記(可能包含背景介紹、痛點描述、期望功能等內容,內容可能零散、口語化、跳著講)。

【你要做的事】
1. 通讀全文,列出客戶提到的每一個痛點或需求。
2. 針對「每一個」痛點,檢查以下三件事有沒有講清楚,沒講清楚就標記為缺漏:
   - 具體環節:這個痛點發生在業務流程的哪一個步驟?(不要只寫「訂單有問題」,要寫清楚「客戶下單後,誰要在什麼時候做什麼動作」)
   - 資料形式:相關的資料現在是用什麼形式存在的?(文字、Excel、紙本、圖片、口頭傳達等)
   - 痛點量化:現在做這件事要花多少時間或人力,客戶希望改善到什麼程度?(現在要多久,想要多久;現在要幾個人,想要幾個人)
3. 針對每一個缺漏,寫出一個「客戶答得出來的具體問題」,不要寫空泛的問題。例如不要寫「你還有其他需求嗎」,要寫「負責人員請假當天如果已經有預約,系統要自動通知客戶改期,還是由行政人員手動處理?」
4. 如果逐字稿裡出現互相矛盾的說法(例如前段說要用 A 方案,後段又提到考慮 B 方案),要特別標記出來,列為最優先要跟客戶確認的項目。

【輸出格式】
用以下 Markdown 格式輸出,不要加其他說明文字:

## 待確認清單

### 痛點 1:[痛點名稱,用客戶的話簡短描述]
- 具體環節:[已知內容,或標註「⚠️ 未提及」]
- 資料形式:[已知內容,或標註「⚠️ 未提及」]
- 痛點量化:[已知內容,或標註「⚠️ 未提及」]
- 追問問題:[針對這個痛點的缺漏,列出 1 到 3 個具體、客戶答得出來的問題]

### 痛點 2:...
(依此類推,每個痛點一個區塊)

### 矛盾或需優先確認事項
[列出逐字稿中互相矛盾、或風險較高、需要優先跟客戶確認的項目,沒有就寫「無」]
```

## 步驟二(對應 Phase 2:AI 落地可能性轉譯)

```
你是一位熟悉「用科技或 AI 解決業務痛點」的顧問。我會給你一份已經確認過的需求清單(通常是步驟一輸出的「待確認清單」,經過跟客戶確認、補齊資訊後的版本),你的任務是幫每一個需求做分類,並附上至少一個類似案例,讓客戶可以用「選擇題」的方式做決定,而不是自己憑空想像解決方案長什麼樣子。

【輸入】
已確認的需求清單(每項需求包含:具體環節、資料形式、量化落差)。

【你要做的事】
1. 針對「每一個」需求,判斷它最接近以下哪一種類型(如果同時符合多種,選最主要的一種,並可以補充次要類型):
   - 自動化:原本要人工做的重複性步驟,改用系統自動處理(例如自動排班、自動發通知、自動核對資料)
   - 生成內容:需要產生文字、圖片、文件等內容(例如自動寫報告初稿、自動生成客服回覆)
   - 數據分析:需要從資料裡面找出趨勢、做統計、做預測
   - 查詢展示:單純是「把資料整理好,讓人可以方便查看或搜尋」,不涉及自動決策或生成
   - 其他:不屬於以上分類的,請具體說明是什麼類型
2. 針對每個分類,找出至少一個類似的產業案例或常見做法(不用是真實公司名稱,可以是「常見於某某產業的做法是...」這種描述),讓客戶知道別人是怎麼處理類似問題的。
3. 說明「為什麼」這個需求適合分類成這一種,用客戶聽得懂的白話,不要用技術術語堆砌。
4. 如果這個需求做起來的技術複雜度偏高、或有已知的常見坑(例如需要串接第三方系統、需要處理隱私資料),要在備註欄提醒。

【輸出格式】
用以下 Markdown 表格輸出,不要加其他說明文字:

## 需求分類表

| 需求 | 分類 | 類似案例 | 為什麼適合這個分類 | 備註(複雜度/風險提醒) |
|---|---|---|---|---|
| [需求名稱] | [自動化/生成內容/數據分析/查詢展示/其他] | [至少一個類似案例的簡短描述] | [白話說明] | [沒有就寫「無」] |
```

## 步驟三(對應 Phase 3:雛形功能分析)

```
你是一位資深技術 PM,擅長比對「已經確認的需求」跟「雛形或 Demo 呈現的內容」,抓出兩者之間對不上的地方。我會給你兩份資料:(1) 已確認的需求清單(通常是步驟一、步驟二輸出並確認過的版本),(2) 雛形或 Demo 的內容。你的任務分兩個階段執行,兩階段之間要停下來讓我確認,不能一次做完就把結果丟出來。

【輸入】
1. 已確認的需求清單(具體環節、資料形式、量化落差、分類)
2. 雛形或 Demo 的內容,可能是以下兩種形式之一(哪一種都可以,不強制要有程式碼):
   - 雛形的實際程式碼(如果剛好有的話)
   - 雛形/Demo 的功能說明、畫面描述、操作流程的文字整理(例如「這個畫面有哪些欄位、按下去會發生什麼事」這種描述)

【你要做的事:階段一】
1. 通讀雛形/Demo 的內容,列出它涵蓋了哪些範圍(畫面、功能、流程)。
2. 針對每一項雛形範圍,標記跟需求清單的相關性:
   - 完全對應需求
   - 部分對應需求
   - 雛形有,但需求清單沒提到
   - 需求清單有,但雛形沒做
3. 做到這裡就停下來,不要往下產出對照表。把範圍清單完整輸出,並附上這句提醒:「這個範圍抓得對嗎?有沒有雛形資訊我遺漏的部分?請確認後我再繼續產出對照表。」

【階段一輸出格式】
用以下 Markdown 格式輸出,不要加其他說明文字:

## 雛形/Demo 範圍清單

| 雛形範圍 | 相關性 | 說明 |
|---|---|---|
| [畫面/功能/流程名稱] | [完全對應需求/部分對應需求/雛形有但需求沒提到/需求有但雛形沒做] | [簡短說明] |

---

【你要做的事:階段二,收到「範圍清單確認沒問題」的回覆後才執行】
針對確認過的範圍,產出以下四張表:

1. 需求覆蓋度對照表:每個需求標記 ✅已實作 / ⚠️部分實作 / ❌未實作 / 🔲僅有畫面沒有邏輯
2. 使用者流程盤點:列出每個使用者流程,標出中斷點:⚠️畫面有但邏輯不明 / ❓需求還沒定案 / 🔴完全沒做(流程正常無中斷則寫「無中斷」)
3. 邊界案例檢查清單:逐一列出待確認清單裡出現過的邊界情況,標記雛形/Demo 有沒有處理到
4. 缺口分析:針對每一個缺口(需求缺口、雛形缺口、兩者不一致的地方),各自回答三個問題:
   - 為什麼這個缺口重要?
   - 不處理會有什麼影響?
   - 建議怎麼解決?

以能讓工程師順利開發為目標。需求描述模糊的地方標記「⚠️ 待補」,不要自己假設答案。如果發現需求之間有矛盾或衝突,要明確指出,不要選一邊自己決定。

【階段二輸出格式】
用以下 Markdown 格式輸出,不要加其他說明文字:

## 需求覆蓋度對照表

| 需求 | 狀態 | 說明 |
|---|---|---|
| [需求名稱] | [✅已實作/⚠️部分實作/❌未實作/🔲僅有畫面沒有邏輯] | [說明] |

## 使用者流程盤點

| 流程 | 中斷點 | 說明 |
|---|---|---|
| [流程名稱] | [⚠️畫面有但邏輯不明/❓需求還沒定案/🔴完全沒做/無中斷] | [說明] |

## 邊界案例檢查清單

| 邊界案例 | 雛形/Demo 是否處理 | 說明 |
|---|---|---|
| [邊界案例] | [有處理/沒處理/不確定] | [說明] |

## 缺口分析

### 缺口:[缺口名稱]
- 類型:[需求缺口/雛形缺口/不一致]
- 為什麼重要:[說明]
- 不處理的影響:[說明]
- 建議解法:[說明]
```

## 步驟四(對應 Phase 4:需求規格書產出)

```
你是一位資深技術 PM,擅長把模糊的業務需求寫成工程師可以直接照做的正式需求規格書。我會給你幾份資料:客戶會議逐字稿或筆記、待確認清單、需求分類表,如果有做過雛形功能分析,也會附上那份結果。你的任務是把這些資料整理成一份完整的需求規格書初稿。

【前置動作:交叉引用探測】
寫任何一個功能模組的完整七個區塊內容之前,先掃描同一份文件裡已經寫好的其他功能模組(如果這是唯一一個功能模組,沒有其他模組可比對,直接跳過這個前置動作,不用產出空清單)。針對正要寫的新模組,列出:
1. 銜接點:這個新模組會用到哪些其他模組已經定義過的資料欄位、狀態、規則
2. 重複定義風險:這個新模組是不是重新定義了其他模組已經定義過的東西
3. 潛在衝突:跟其他模組的規則有沒有矛盾

把這份銜接清單先輸出。如果是一步一步對話式產出(每次只寫一個模組),銜接清單輸出後提醒:「先確認這份銜接清單,再繼續往下寫這個模組的完整內容。」如果是一次要產出多個模組,銜接清單產出後就直接依此原則往下寫完所有模組,已經定義過的規則用「參見 OO 模組」引用,不要重複寫一次。

【前置動作輸出格式】
## 交叉引用探測:[模組名稱]

| 項目 | 內容 |
|---|---|
| 銜接點 | [用到哪些其他模組的欄位/狀態/規則,沒有就寫「無」] |
| 重複定義風險 | [有沒有重新定義其他模組已定義的東西,沒有就寫「無」] |
| 潛在衝突 | [跟其他模組規則有沒有矛盾,沒有就寫「無」] |

---

【輸入】
1. 客戶會議逐字稿或筆記
2. 已確認的待確認清單(步驟一輸出並確認過的版本)
3. 需求分類表(步驟二輸出的版本)
4. 雛形功能分析結果(步驟三輸出的版本,若這次沒有雛形/Demo 可比對,略過這份資料即可)

【你要做的事】
針對每一個功能模組,產出以下七個區塊的內容。「邊界例外」這個區塊如果想不出來、或資訊不足以判斷,絕對不能省略或留空,必須明確寫「⚠️ 待補:[需要跟客戶確認的具體問題]」,不能假裝沒這回事直接跳過。

1. 痛點總結:這個功能要解決客戶的什麼問題,原本是怎麼做的,改善後預期效益是什麼
2. 業務邏輯:這個功能實際運作的規則是什麼(誰可以做什麼、什麼條件下會發生什麼事、有沒有例外規則)
3. User Story:用「身為[角色],我想要[做什麼],以便[達成什麼目的]」的格式,列出這個功能涉及的所有使用者故事
4. 驗收標準(Acceptance Criteria):每個 User Story 對應的具體驗收條件,要寫成「在什麼情況下,應該發生什麼結果」,工程師看了就知道怎麼判斷這功能有沒有做對
5. 資料規格:這個功能涉及哪些資料欄位、欄位的資料型態、是否必填、資料從哪裡來、要保存多久
6. 狀態流程:如果這個功能涉及「狀態轉換」(例如預約從「待確認」到「已確認」到「已完成」),條列所有狀態與轉換條件
7. 邊界例外:列出所有「不是正常流程,但實際可能發生」的情況,以及系統該怎麼處理。想不到具體答案的項目,一律標注「⚠️ 待補」,不可省略這個項目本身。

【輸出格式】
用以下 Markdown 格式,針對「每一個」功能模組各輸出一個完整區塊:

## 功能模組:[模組名稱]

### 痛點總結
[內容]

### 業務邏輯
[內容]

### User Story
- 身為[角色],我想要[做什麼],以便[目的]

### 驗收標準(Acceptance Criteria)
- [對應的 User Story] 在[情境]下,應該[結果]

### 資料規格
| 欄位 | 型態 | 必填 | 說明 |
|---|---|---|---|
| ... | ... | ... | ... |

### 狀態流程
[條列或用文字描述狀態轉換,沒有狀態流程的功能寫「不適用」]

### 邊界例外
- [情況]:[處理方式,或「⚠️ 待補:[具體問題]」]

(每個功能模組都重複以上結構)
```

---

## 提醒:這只是初稿

這四段 Prompt 產出的東西都只是初稿,尤其是步驟四的規格書。PM 拿到初稿後,仍要做兩件事:跟客戶對齊(確認有沒有理解錯、有沒有漏掉)、跟 AI 一起補邊界案例並比對業界慣例。這個二次收斂的動作,AI 不能取代,一定要 PM 親自跑完。
