# PM-SDD 工作模式:一人接案的需求釐清到驗收 SOP

## 這是什麼、要解決什麼問題

你是不是也遇過:客戶說「我要一個可以幫我管理預約的系統」,工程師聽完就去刻了一個陽春的月曆,結果做完客戶才說「不對,我要的其實是每個服務人員之間的個案資料要互相隔離,而且客戶約完一次之後,之後都只能約回同一個人」(這是示意情境,用來說明「藏在對話裡、客戶自己講的時候都沒意識到要講清楚」的細節長什麼樣子,不是逐字重現某次真實對話)。這種細節,才是整個案子做不做得成、會不會被客戶罵的關鍵。

這套 SOP 要處理的,是「翻譯官」的角色問題。PM 是客戶(講需求、講痛點、講預算)跟工程師(要精確規則、要可以直接照做的規格)之間的翻譯官。AI 可以幫翻譯官先打草稿、先抓漏,但最後拍板定稿、跟客戶對過、確保工程師沒有自己腦補的部分,還是要 PM 出面扛。

背後的團隊分工是「1+1 團隊模式」:一個 PM 負責搞懂客戶要什麼、把需求變成規格,搭配一個(通常是外包的)全端工程師負責把規格變成能動的系統。PM 不用會寫代碼,工程師也不用自己猜需求,兩邊各司其職。這份文件把整個流程拆成 7 個階段(Phase 0 到 Phase 6),適合任何走接案模式、團隊組成類似「PM×1 + RD×1」的工作者參考。

適用對象:任何要接案子的 PM,包含完全沒用過 AI 工具的人。文件裡第一次出現業界術語時會用括號輕輕帶過,完整解釋看另一份《PM-SDD 術語對照表》。

在正式開始跑這套 SOP 之前,建議先用《PM-SDD 團隊導入健檢清單》(`docs/04-團隊導入健檢清單.md`)快速確認這個案子、這個 PM+RD 搭配適不適合走 1+1 SDD 模式,再決定要不要往下走完整 6 個 Phase。

## 六個階段怎麼走

### Phase 0:客戶初談(先把客戶的話聽完,不要急著下結論)

**在做什麼**:開會(現場或線上都可以),把客戶講的話全部錄音,事後轉成逐字稿。

**具體怎麼做**:
- 約時間開會,錄音全程進行
- 會後把錄音轉成逐字稿(文字檔)
- 先不要在會議中急著下結論或承諾做法,重點是讓客戶把話講完

**完成的判斷標準**:逐字稿產出完整,涵蓋客戶提到的背景、痛點、限制條件(例如指定的技術、預算、時程),沒有明顯漏掉整段沒錄到的內容。

---

### Phase 1:場景引導(抓出客戶自己都沒講清楚的地方)

**在做什麼**:讓 AI 通讀 Phase 0 的逐字稿,反過來抓「客戶講得模糊、還沒講到的地方」,整理成一張「要跟客戶確認的清單」。這步驟業界的說法接近「需求澄清」,還不到正式規格書的程度。

**具體怎麼做**:
- 把逐字稿丟給 AI(用《AI 協作提示詞:需求釐清四步驟》步驟一 Prompt)
- AI 針對每個痛點,標出三件事:具體環節(發生在業務流程的哪一步)、資料形式(文字/圖片/Excel/紙本)、痛點量化(現在要花多久 → 客戶想要多久)
- 沒講清楚的地方,AI 會列成「客戶答得出來的具體問題」,不是空泛的「你還有其他需求嗎」

**完成的判斷標準**:每個痛點都標好這三個標籤,而且列出的追問問題夠具體,客戶看了知道要怎麼回答,不是需要再猜一次的模糊問題。

---

### Phase 2:AI 落地可能性轉譯(把需求變成選擇題,不要讓客戶自己空想解法)

**在做什麼**:針對 Phase 1 確認過的每個需求,AI 幫忙分類(是要自動化、產生內容、還是做數據分析查詢),並附上至少一個類似案例,讓客戶用「選這個還是那個」的方式做決定。

**具體怎麼做**:
- 用《AI 協作提示詞:需求釐清四步驟》步驟二 Prompt,對每個需求輸出分類、類似案例、為什麼適合這個分類
- 把分類表拿給客戶看,讓客戶確認或調整,而不是 PM 自己單方面決定方向

**完成的判斷標準**:每項需求都有分類、至少一個案例,分類理由客戶一看就懂為什麼是這樣分,客戶有明確表態(同意或提出調整),不是被動接受。

---

### Phase 3:雛形對齊(先讓客戶「看到」東西,不是先讓他「想像」東西)

**在做什麼**:做一個看得到、點得到的 Demo 或畫面草稿(不用是能上線的完整系統),給客戶確認方向對不對。這種「先做雛形給客戶看」的手法,業界叫 Vibe Coding,只是原本容易停在這裡(雛形做完就結案),這套 SOP 要求雛形之後一定要接著寫成正式規格書,不能雛形做完就算了。

**具體怎麼做**:
- 依照 Phase 1、Phase 2 收斂出來的需求方向,做出前端 Demo 框架或畫面草稿
- 拿給客戶看,請客戶明確確認「對,就是這樣」,或提出具體修改意見

**完成的判斷標準**:客戶看過 Demo 後給出明確回饋(不是「好像可以」這種模糊回應)。而且很重要:這個雛形不是用完就丟的東西,它是下一步規格書的骨架來源,雛形上呈現的每個畫面、欄位、流程,都要能對應到規格書的內容。

現在有對應的 AI prompt 可以用(步驟三:雛形功能分析,見《AI 協作提示詞:需求釐清四步驟》),讓 AI 比對雛形/Demo 涵蓋的範圍跟已確認需求清單,抓出「可以直接用、還要新做、被漏掉沒做進雛形」的落差,產出對照表跟缺口分析,PM 拿這份結果去跟客戶或工程師對焦,不用自己肉眼比對。

**案例示範(虛構情境,示範用)**:以下用一個虛構的接案情境示範這個 Phase 實際怎麼跑。這個情境裡團隊的做法本來就是先出前端 Demo 框架給客戶確認,再進後端開發,剛好完全對應這個 Phase,不用額外調整流程。

---

### Phase 4:需求規格書產出 + 二次收斂(寫成工程師可以直接照做的說明書,而且要來回校對兩次)

**在做什麼**:用 AI 把前面蒐集到的痛點、分類結果、雛形回饋,整理成一份正式的需求規格書初稿(業界叫 FRD,功能需求規格書,細節看術語對照表),再由 PM 帶著跑兩輪收斂。

**具體怎麼做**:
- 用《AI 協作提示詞:需求釐清四步驟》步驟四 Prompt,產出初稿,內容包含:痛點總結、業務邏輯、User Story(使用者故事)、驗收標準、資料規格、狀態流程、邊界例外
- 收斂第一輪:拿初稿去跟客戶對,確認有沒有理解錯、有沒有漏掉
- 收斂第二輪:PM 跟 AI 一起補「邊界例外」(例如「負責人員請假當天已經有預約,系統要怎麼處理」這種情況),並跟業界慣例比對,確認有沒有明顯遺漏

如果案子有多個功能模組,FRD 產出時 AI 會先做「交叉引用探測」,檢查新模組跟其他已寫好模組有沒有銜接點或衝突,避免模組互相打架、規則重複定義。

**完成的判斷標準**:
- 邊界例外欄位不能空白,想不到答案也要標注「⚠️ 待補」
- 兩輪收斂都要跑完,不能只跑一次就交出去
- 這份文件變成全案「唯一標準」,之後工程師、業主、PM 看的都是同一份,不能各自理解各自的版本

**案例示範(虛構情境,示範用)**:以下用一個虛構的接案情境示範這個 Phase 實際怎麼跑。客戶方原本就準備了一份 RFQ(報價需求書,細節見術語對照表)。這份文件本身是客戶為了報價、發包而整理的需求文件,嚴格來說還不是 FRD,但內容已經有一定細節,可以當作 FRD 的雛形來延伸補強,這是 PM 這邊的判斷、不是客戶說這份文件就是 FRD。收斂階段留下來的待確認清單,常見的例子像是「討論通知服務要用哪一種比較划算」「前端 Demo 的實際範圍與時程」「報價是否已經回覆客戶」這類問題,就是收斂還沒跑完的具體例子,對照的正是這個 Phase 該收斂掉的東西。

---

### Phase 5:交接外包工程師(把規格書交出去,PM 負責盯有沒有做對,不是自己教工程師寫程式)

**在做什麼**:規格書定稿後,接下來「怎麼蓋」是工程師的專業,PM 不用插手。工程師會自己規劃技術做法、寫測試計畫、動手開發,最後上線前把整套系統跑一遍確認沒漏東西(這三件事業界分別叫 Tech Spec、Test Spec、SIT,詳細定義看《PM-SDD 術語對照表》)。PM 全程不寫技術方案,只負責檢查工程師有沒有把規格書的每一條都做到。

**具體怎麼做**:
- 把定稿的 FRD 交給工程師
- 工程師產出技術規格與測試計畫,開始寫程式
- PM 定期用 FRD 逐條核對進度,發現工程師理解偏了要馬上抓回來對齊,不能等到最後才發現

**完成的判斷標準**:每個 FRD 條目都能在工程師的實作或測試裡找到對應的東西,沒有「規格書沒寫,但工程師自己加上去」或「規格書有寫,工程師卻漏做」的情況。

**案例示範(虛構情境,示範用)**:以下用一個虛構的接案情境示範這個 Phase 實際怎麼跑。在這個情境裡,案子走到雛形對齊之後、還沒進入這個階段,所以這裡先不編造細節,實際案子推進到這一步時,重點就是回到上面「具體怎麼做」段落說的三件事:交規格、盯進度、發現偏差馬上拉回來對齊。

---

### Phase 6:驗收(跟客戶一起確認東西真的能用,不是關起門自己測完就交差)

**在做什麼**:系統做完之後,拉著客戶一起驗收,不是內部工程師自己測過就算數。

**具體怎麼做**:
- 依照 FRD 的驗收標準,一條一條讓客戶實際操作、確認
- 有問題當場記錄,分成「這次上線前一定要修」跟「之後再排」兩類

**完成的判斷標準**:客戶親自確認過每個 User Story 都能實際跑通,不是 PM 或工程師代替客戶說「應該沒問題」。

**案例示範(虛構情境,示範用)**:以下用一個虛構的接案情境示範這個 Phase 實際怎麼跑。同 Phase 5,這個情境還沒走到這一步,實際案子跑到這裡時,就是照上面「具體怎麼做」段落逐條驗收、當場分類問題的輕重緩急。

## 風險對策表(對照過去真實踩過的坑)

| 風險點 | 白話講的踩坑對照 | 對策 |
|---|---|---|
| Phase 2 分類做太快,沒真的讓客戶選 | 之前有團隊只出「畫面看起來對」的東西,沒真的挖客戶的業務邏輯,結果上線後才發現漏洞 | 每個需求分類後一定要附至少一個類似案例,讓客戶真的做選擇,不是 PM 單方面決定 |
| Phase 4 只收斂一次就交出去 | 曾經有案子裡,PM 主導的規格段落寫得太粗,規格數量最後暴增到收不了尾 | 兩輪收斂是強制步驟,不能省 |
| 外包工程師拿到 FRD 後自己腦補 | 曾經有案子裡,工程師人一多,大家各自照自己理解做事,規格書變得不是唯一標準 | FRD 模板強制要填「邊界例外」欄位,沒填等於這份文件還沒做完 |

## 結尾

這份 SOP 定期跟著實戰案例更新,不是一次寫死。每跑完一個案子,發現哪個 Phase 卡住、哪個判斷標準不夠明確,就回來改這份文件。
