# PM-SDD Workflow

一套給接案 PM(產品經理)用的需求釐清與規格驅動開發(SDD, Spec-Driven Development)工作方法論,含一份可直接在 Claude Code 裡使用的 `/需求轉譯` skill。

## 解決什麼問題

接案或外包合作最容易出事的地方,不是工程師寫不出程式,而是「客戶講的話」跟「工程師照做的東西」中間出現落差。客戶說「我要一個可以幫我管理預約的系統」,工程師聽完就去刻了一個陽春的月曆,結果做完客戶才發現「不對,我要的其實是更細的業務規則」。這種藏在對話裡、客戶自己講的時候都沒意識到要講清楚的細節,才是整個案子做不做得成的關鍵。

這套方法論把 PM 定位成客戶跟工程師之間的「翻譯官」:AI 可以幫忙先打草稿、先抓漏,但最後拍板定稿、跟客戶對過、確保工程師沒有自己腦補的部分,還是要 PM 出面扛。背後假設的團隊組成是「1+1 團隊模式」:一個 PM 負責搞懂客戶要什麼、把需求變成規格,搭配一個(通常是外包的)全端工程師負責把規格變成能動的系統。PM 不用會寫代碼,工程師也不用自己猜需求。

## 怎麼使用

依你的身份跟工具,有三種切入方式:

**人類讀者**:先讀 `docs/01-工作模式.md`,這是白話寫的 SOP,說明從客戶初談到驗收共 7 個階段(Phase 0 到 Phase 6)怎麼走,每個階段在做什麼、具體怎麼做、完成的判斷標準是什麼。不熟業界術語的話,對照 `docs/03-術語對照表.md` 查。開始一個新案子前,先過一次 `docs/04-團隊導入健檢清單.md`,確認這個案子跟這個工程師搭配適不適合走這套流程。

**任何對話式 AI 的使用者**(ChatGPT、Claude、Gemini 等):直接用 `docs/02-AI協作提示詞-需求釐清四步驟.md` 裡的四段完整 Prompt,依序複製貼上執行。四段 Prompt 分別對應 SOP 的 Phase 1 到 Phase 4:場景引導、AI 落地可能性轉譯、雛形功能分析、需求規格書(FRD)產出。

**Claude Code 使用者**:直接呼叫 `/需求轉譯` skill,把客戶會議逐字稿或筆記交給它,它會依照四步驟的節奏,每步驟停下來等你確認,最後產出一份 FRD(功能需求規格書)初稿。細節見 `.claude/skills/需求轉譯/SKILL.md`。

## 目錄結構

```

  01-工作模式.md                          給人讀的 SOP 主文件,7 個 Phase 的完整說明

  03-術語對照表.md                        業界術語的白話翻譯對照

  knowledge/輸入處理說明.md                Claude Code 環境的輸入判斷細節
```

## 這套方法論的來源

這份方法論整合了兩個來源:

1. DevOpsDays Taiwan 2026 講者鍾筑安(Judych)分享的「從 Vibe Coding 走向 SDD」1+1 團隊方法論,啟發了這裡的團隊分工模式與 Phase 3(雛形對齊)之後「雛形不能做完就結案,一定要接續寫成正式規格書」的核心主張。
2. 作者自己在接案實務中累積的客戶需求釐清框架,包含場景引導、需求分類、雛形功能分析、規格書兩輪收斂等具體步驟設計。

案例段落裡出現的情境皆為示範用的虛構情境,不對應任何特定客戶或真實逐字稿。
