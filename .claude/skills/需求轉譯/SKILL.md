# 需求轉譯

把客戶會議逐字稿或筆記,依照「場景引導 → AI 落地可能性轉譯 → 雛形功能分析 → FRD 產出」四步驟,轉成需求規格書(FRD)初稿。完整方法論看 `docs/01-工作模式.md`,術語對照看 `docs/03-術語對照表.md`,Claude Code 環境的輸入判斷細節看 `knowledge/輸入處理說明.md`。

這是**草案版本**。目前流程在單一對話裡手動帶過四步,每步都要停下來給使用者確認才能繼續。之後流程熟了,可以考慮收斂進其他 SDD 工作流工具;現階段先用這個薄殼版本跑。

## 輸入

**必要**:客戶會議逐字稿或筆記,兩種給法都可以:
- 檔案路徑:`/需求轉譯 docs/逐字稿.md`
- 直接貼文字:`/需求轉譯` 後面接整段文字

**選用**:雛形或 Demo 資料(程式碼、截圖說明、功能清單皆可),用於步驟 3,沒有就跳過該步驟,不影響其他步驟。

兩者(檔案路徑、貼上文字)都沒附時,不會自己亂猜要分析哪份資料,會先停下來問清楚。

## GATE:對焦(不可跳過)

收到任務後,第一步:
1. 複述一句「我理解你要把 [輸入來源] 轉成 FRD 初稿」
2. 依 `knowledge/輸入處理說明.md` 判斷輸入是檔案路徑還是貼上的文字,兩者都沒有就先問使用者要分析哪份資料
3. 問「方向對嗎?」→ 使用者說 OK 才往下

## 流程

### 步驟 1:場景引導

Read `docs/02-AI協作提示詞-需求釐清四步驟.md` 執行「步驟一」,輸入用取得的逐字稿或筆記,產出「待確認清單」。

> **強制停止點:待確認清單產出後,完整呈現給使用者,問「這份待確認清單有沒有要調整?確認後才進步驟 2。」等使用者明確回覆才繼續。**

### 步驟 2:AI 落地可能性轉譯

Read `docs/02-AI協作提示詞-需求釐清四步驟.md` 執行「步驟二」,輸入用確認過的待確認清單,產出「需求分類表」。

> **強制停止點:需求分類表產出後,完整呈現給使用者,問「這份分類對不對?確認後才進步驟 3。」等使用者明確回覆才繼續。**

### 步驟 3:雛形功能分析

Read `docs/02-AI協作提示詞-需求釐清四步驟.md` 執行「步驟三」。先問使用者有沒有雛形或 Demo 資料(程式碼或功能/畫面描述皆可),沒有就跳過這步直接進步驟 4。

這步驟本身分兩階段:階段一(雛形範圍清單)產出後**必須停下來**等使用者確認,才能執行階段二(四張對照表)。

> **強制停止點:範圍清單產出後,問「這個範圍抓得對嗎?有沒有雛形資訊我遺漏的部分?」等使用者明確回覆才進階段二。**

階段二的四張對照表(需求覆蓋度對照表/使用者流程盤點/邊界案例檢查清單/缺口分析)產出後,同樣**必須停下來**等使用者確認,才能進步驟 4。

> **強制停止點:四張對照表產出後,完整呈現給使用者,問「這份雛形分析對不對?確認後才進步驟 4。」等使用者明確回覆才繼續。**

### 步驟 4:FRD 產出

Read `docs/02-AI協作提示詞-需求釐清四步驟.md` 執行「步驟四」,輸入用逐字稿/筆記、確認過的待確認清單、需求分類表,以及雛形功能分析結果(如果有做步驟 3)。若這次規格書有多個功能模組,先產出前置動作的交叉引用探測清單再往下寫。

輸出後提醒使用者:「這是 AI 產出的初稿,『邊界例外』欄位標 ⚠️ 待補的地方需要你自己補,還要跑兩輪收斂(跟客戶對齊、補邊界案例)才能定稿,這個 skill 不會自動幫你做收斂。」

### 步驟 5:品質檢查(必須執行)

- [ ] 輸出格式符合四步驟文件各步驟的【輸出格式】規定
- [ ] 無遺漏欄位(尤其「邊界例外」不可空白,想不到答案要標 ⚠️ 待補)
- [ ] 邊界情況已處理或明確標注待補

## 輸出

各階段產出都用 Markdown 呈現在對話中,不主動寫檔案。使用者要求存檔才問清楚路徑(通常是對應客戶專案資料夾,或版控裡的變更紀錄資料夾)。

## Guardrails

- 四個步驟之間**必須**停下來等使用者確認,禁止一次跑完就把 FRD 初稿丟出來。
- 步驟 3 內部的階段一、階段二之間也要停下來確認,不可一次做完。
- FRD 初稿的「邊界例外」欄位空白或想不到答案時,必須標注「⚠️ 待補」,不可省略或假裝沒有邊界情況。
- 這個 skill 只做「初稿產出」,不做二次收斂;二次收斂是 PM 的工作,不是這個 skill 的責任範圍。
- 找不到 `docs/02-AI協作提示詞-需求釐清四步驟.md` 時直接停止並告知,不要憑記憶自己編一套流程。

## 完成條件(全部達到才算完成)

- [ ] GATE 已通過
- [ ] 四個步驟(含步驟 3 的兩階段)每個停止點都已等使用者確認
- [ ] 品質檢查已通過
- [ ] 使用者已確認 FRD 初稿內容
